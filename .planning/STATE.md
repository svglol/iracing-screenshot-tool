Carry-forward lesson: after any mechanical `replace_all` on an API migration, run the downstream verifier (test runner, linter, compiler) — don't trust grep-count alone. For multi-line patterns, a second `replace_all` with the newline+tab explicitly in the pattern catches the chained forms. Alternative future pattern: run prettier FIRST to collapse chains to single-line, THEN do the replace_all on a normalized file.
- **2-commit bisect shape achievable without interactive staging via revert-and-reapply** (v2.0 Phase 10 Plan 01) — The plan offered a minimum-scope derogation to a single combined commit if `git add -p` became fragile. Instead, after Tasks 1+4 both landed in the working tree (deps swap + scripts rewire both touching package.json), the deps-only commit 1 was staged by temporarily reverting the scripts portion back to the jest-era form (keeping Task 1's devDeps-removal + jest-block-removal), committing, then re-editing scripts back to the Vitest form for commit 2. No interactive TTY needed — works in fully non-interactive harness environments. Pattern is a carry-forward for future bisect-shape preservation when two semantic changes land in adjacent regions of a single file.
- **Vitest runtime improvement over Jest baseline** (v2.0 Phase 10 Plan 01) — `npm test` on the 5 root test files runs in ~220 ms under Vitest 4.1.5 vs ~0.6 s under Jest 30 (measured in Phase 10 CONTEXT — "5 test files, 256 tests, 0 snapshots, ~0.6s runtime under Jest 30"). ~3× speedup attributable to Vite's native ESM pipeline + no babel-jest transform + v8 native coverage vs istanbul's AST-based instrumentation. Not a deliberate performance optimization — falls out of the runner swap.
- **Node's `require` cannot resolve `.ts` extensions; Vite's resolver can via `import`** (v2.0 Phase 12 Plan 01) — When Vitest loads a `.test.ts` file, its Vite transformer converts `import` statements via Vite's resolver (which handles `.ts`, `.tsx`, etc.), but leaves `require(...)` calls in-place for Node's native resolver (which only handles `.js/.json/.node`). After renaming `.js → .ts` in `src/utilities/`, any consumer using extensionless `require('./name')` breaks at test-time. Fix patterns: (a) switch test imports to `import` form (D-12-09 permits this as the effective path change); (b) hoist dynamic function-body requires to top-of-file and add explicit `.ts` extension (Vitest's transformer statically detects top-of-file requires even if it leaves them CJS at runtime); (c) do not switch to `import * as fs from 'fs'` in source files whose test spies on fs — ESM namespaces are sealed and `vi.spyOn` cannot redefine their properties. Source+test must share the same module instance via `const fs = require('fs')` on both sides.
- **`@ts-expect-error` for Electron-injected runtime globals** (v2.0 Phase 12 Plan 01) — `process.type` is injected by Electron at runtime in both main + renderer processes, but is NOT present in `@types/node`'s `Process` interface (node doesn't know about Electron). Any code that branches on `process.type === 'renderer'` (config.ts dual-surface, logger.ts lazy `require('electron')`) needs `// @ts-expect-error — process.type is Electron-injected at runtime; not in @types/node Process`. When we install `@types/electron` (v2.1 candidate or whenever a plan needs it), these retire in one pass.
- **tsconfig `types:[...]` + `skipLibCheck:true` as counterweight to renaming tests into scope** (v2.0 Phase 12 Plan 01) — When `src/utilities/*.test.js` became `*.test.ts`, the existing `include: ["src/utilities"]` swept them into tsc scope. Tests use Vitest globals (`describe`, `test`, `expect`, `vi`) which come from `vitest/globals`. The explicit `types: ["node", "vitest/globals"]` array pins the wanted ambient types AND blocks auto-inclusion of random `@types/*` packages. `skipLibCheck: true` insulates our source from third-party `.d.ts` subpath-import quirks (e.g., vite's `#types/hmrPayload`, @vitest/expect's `@vitest/utils/display`) — these are known upstream issues, not our problem. Both flags are widely standard in Vite-based projects and non-controversial.
- **`export function X()` over `module.exports = {X}` in TS** (v2.0 Phase 12 Plan 01) — Using `module.exports = {...}` without any `export` or `import` keyword makes a `.ts` file a SCRIPT (global scope), not a MODULE. tsc reports `TS6200 Definitions conflict` / `TS2451 Cannot redeclare` cross-file when two scripts declare same-named identifiers. Fix: use `export function ...` native ES syntax in all source files. Vite (esbuild) emits CJS `exports.X = X` at build time under its default module format, preserving the `const {X} = require('./name')` consumer pattern on the main-process side. `config.ts` is the one exception: its dual-surface `if/else module.exports = {...}` assignment is legal TS-on-CJS (`allowJs` + CommonJS module default) but needs one `@ts-expect-error` on the outer `process.type` guard.
