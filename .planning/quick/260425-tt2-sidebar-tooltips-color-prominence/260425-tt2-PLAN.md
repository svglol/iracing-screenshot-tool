<objective>
Restore the colored, low-key tooltip aesthetic in `SideBar.vue` that the v2.0 Buefyâ†’Oruga / Bulma 0.9â†’1.0 migration regressed.

Three coupled regressions to fix in one shot:

1. **Lost color information** â€” every `<o-notification>` carries an inline `color: yellow` override, and the trailing `<style>` block targets `.message.is-warning .message-body` / `.message.is-info .message-body` (Buefy/Bulma 0.9 message-component DOM that Oruga 0.13 + Bulma 1.0 no longer emit â€” Oruga theme-bulma now renders `.notification > .media > .media-content`, verified in `node_modules/@oruga-ui/theme-bulma/dist/theme.js` `notification:` config). Result: warning, info, and danger panels all render with the same hardcoded yellow inline rule, the variant-specific selectors are dead code, and the user reads the same color regardless of meaning.

2. **Way too prominent** â€” the message bodies are wrapped in `<strong>...</strong>` and Bulma 1.0's `.notification` defaults give the panel a fairly heavy padding + font-size. The user wants them less shouty.

3. **Wide interlines** â€” Bulma 1.0's `.notification` inherits a generous line-height. Tighten it.

Constraint: the SideBar `<style>` block is NOT scoped (and history shows it has been unscoped for a while â€” `.control-label { font-weight: 700 }` lives there too as an app-wide rule). Programmatic toasts fired from `useOruga().notification.open(...)` for screenshot success/failure also render to `.notification` and would inherit any global override. To avoid touching them, namespace ALL new rules behind a `.sidebar-tooltip` class added to the four in-template notifications. Toasts (which are mounted into a `.notices` container by Oruga, not in this template) won't carry that class.

Output: edits to `src/renderer/components/SideBar.vue` only.
</objective>
