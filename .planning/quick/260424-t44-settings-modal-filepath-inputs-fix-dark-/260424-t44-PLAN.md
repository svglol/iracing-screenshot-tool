<objective>
Fix two coupled v2.0 visual regressions in SettingsModal.vue filepath rows:

1. **Dark-on-dark text:** the three disabled filepath inputs (Screenshot Folder, Screenshot Keybind, Reshade INI) display their `:value` in near-invisible text because Bulma 1.0's dark-scheme `--bulma-input-disabled-color` resolves to `var(--bulma-text-weak)` (grey) on `var(--bulma-background)` (~9% lightness). Our current main.scss rule `input, select, textarea { color: var(--bulma-input-color, inherit); }` is a no-op: Bulma 1.0 does not emit `--bulma-input-color` as a variable — the color is composed at runtime as `hsl(var(--bulma-input-h), var(--bulma-input-s), var(--bulma-input-color-l))`. So the fallback fires, `inherit` gets the parent gradient's inherited color, and the `[disabled]` selector in Bulma's input mixin still wins with `color: var(--bulma-input-disabled-color)`. We need to override `--bulma-input-disabled-color` at `:root`.

2. **Buttons floating outside inputs:** the three filepath rows use `<o-field class="settings-inline-field">` without the Oruga `addons` prop, so the wrapping `<div class="field">` does NOT get Bulma's `is-addons` class, and the input and button render as two independent pills with rounded borders on both sides and a visible gap. The markup is already structurally correct (sibling `<o-input expanded>` + `<p class="control"><o-button>`) — we just need to flip the `addons` prop on. Additionally, `.settings-action { width: 130px }` pushes the button past its content box, which visually amplifies the detachment; drop it so the button sizes to its text and the shared-radius addon seam renders cleanly.

Purpose: restore v1.4-equivalent legibility and the compact addon widget look the user expects from a filepath picker.

Output: edits to main.scss (2-3 line block swap) and SettingsModal.vue (add `addons` on 3 `<o-field>` wrappers, drop 1 width rule). No new files.
</objective>

Bulma 1.0 input CSS variable surface (node_modules/bulma/sass/form/shared.scss + themes/dark.scss):
- `--bulma-input-h`, `--bulma-input-s`, `--bulma-input-l` — HSL triplet for non-disabled input background
- `--bulma-input-color-l` — lightness for non-disabled text color (default `var(--bulma-text-strong-l)`)
- `--bulma-input-disabled-color` — text color for [disabled] inputs (default `var(--bulma-text-weak)` — THIS IS WHAT WE OVERRIDE)
- `--bulma-input-disabled-background-color` — bg for [disabled] inputs (default `var(--bulma-background)` — dark gray in dark scheme, OK contrast with readable text, but we can bump one shade if visual separation suffers)
- `--bulma-input-disabled-border-color` — border for [disabled] inputs (default `var(--bulma-background)`)

NOTE: `--bulma-input-color` does NOT exist as a resolvable CSS variable. The mixin at bulma/sass/form/shared.scss:118-122 composes it inline: `color: hsl(var(--bulma-input-h), var(--bulma-input-s), var(--bulma-input-color-l));`. Our existing main.scss fallback is writing to a variable nothing reads.

Dark-scheme values from bulma/sass/themes/dark.scss (enabled by our `:root { color-scheme: dark }` lock):
- `--bulma-scheme-main-l: 9%` (input bg base)
- `--bulma-text-weak-l` computed to ~55% → `--bulma-text-weak` is mid-grey
- Mid-grey text on dark-9% background = ~1.8:1 contrast → below WCAG, reads as invisible for a glance
