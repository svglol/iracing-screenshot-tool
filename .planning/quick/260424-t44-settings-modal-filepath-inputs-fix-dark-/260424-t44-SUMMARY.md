# Quick Task 260424-t44: SettingsModal filepath inputs — dark contrast + addon layout

One-liner: Override Bulma 1.0 disabled-input CSS vars for readable light-grey text on dark bg; add `addons` prop to three filepath `<o-field>` wrappers for joined input+button widget.

## Root Causes

### Contrast bug (T44-UI-01)

Bulma 1.0 does NOT emit `--bulma-input-color` as a resolvable CSS custom property. The color for non-disabled inputs is composed inline at the mixin level as `hsl(var(--bulma-input-h), var(--bulma-input-s), var(--bulma-input-color-l))` — no single variable to override. The existing `main.scss` block `input, select, textarea { color: var(--bulma-input-color, inherit); }` therefore always resolved to the `inherit` fallback and had zero effect on any input's color.

For **disabled** inputs, Bulma 1.0's `[disabled]` selector applies `color: var(--bulma-input-disabled-color)`, which defaults to `var(--bulma-text-weak)`. In our `:root { color-scheme: dark }` lock, `--bulma-text-weak` resolves to mid-grey (~55% lightness) on a `--bulma-scheme-main-l: 9%` background — approximately 1.8:1 contrast ratio, visually invisible at a glance.

**Fix:** Add three CSS custom property overrides to the existing `:root` block in `main.scss`:
- `--bulma-input-disabled-color: hsl(0, 0%, 85%)` — light-grey text
- `--bulma-input-disabled-background-color: hsl(0, 0%, 18%)` — slightly lighter than the raw scheme background for visual separation
- `--bulma-input-disabled-border-color: hsl(0, 0%, 28%)` — visible border

Also deleted the no-op `input, select, textarea { color: var(--bulma-input-color, inherit); }` block — it referenced a non-existent variable and its documentation was actively misleading.

### Button detachment bug (T44-UI-02)

The three filepath rows in `SettingsModal.vue` used `<o-field class="settings-inline-field">` without the Oruga `addons` boolean prop. Without `addons`, Oruga's Field component does not apply Bulma's `has-addons` class to the wrapper `<div class="field">`, so the contained `<o-input>` and `<p class="control"><o-button>` render as independent flex children with full rounded borders on both sides and a visible gap between them.

The existing `.settings-action { width: 130px }` desktop rule compounded the visual break by pushing the button past its content box, creating an additional dead zone between the input edge and the button text.

**Fix:** Add `addons` to the three filepath `<o-field>` wrappers (lines 18, 36, 248). Remove the desktop `.settings-action { width: 130px }` rule so the button sizes to its label text. The mobile `@media (max-width: 900px) { .settings-action { width: 100% } }` rule is retained unchanged.
