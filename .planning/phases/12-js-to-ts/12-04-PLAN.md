<objective>
Convert the 10 Vue Single File Components (SFCs) from `<script>` to `<script lang="ts">`. This is a mechanical block-tag change — templates and styles are UNCHANGED; only the script block attribute gains `lang="ts"` and (where tractable) minimal type annotations are added inside the script body. No Composition API rewrite, no `defineComponent` import unless a type error forces it (D-12-08), no `<script setup>`.

**Why SFCs fourth (D-12-02):** Converting after main (Plan 02) and renderer entry (Plan 03) means the main.ts/router/index.ts files that consume these SFCs are already typed. Plan 04 just flips the script block and lets Vite's `@vitejs/plugin-vue` pick up the change — plugin-vue already handles `lang="ts"` out of the box (confirmed via the current devDep `@vitejs/plugin-vue: ^6.0.6`).

**Why before Plan 05 (tooling expansion):** Getting SFCs into `.ts`-mode lets Plan 05 install `vue-tsc` and wire it into the `type-check` script. If SFCs were still `.vue` with `<script>` at Plan 05 time, vue-tsc would false-report all script blocks as implicit `any`.

**Why NOT one SFC per plan:** Mechanical scope is tiny per file (add 5 characters: ` lang="ts"` to one tag). Ten SFCs fit in one plan under context budget because the change is so small.

**Scope explicit exclusions:**
- No Composition API rewrites
- No `<script setup>` conversion
- No template changes (v-if, v-for, refs, events — all untouched)
- No style block changes
- No new type definitions beyond what's needed to silence type errors inside the converted script bodies
- No `defineComponent` wrapper unless a specific SFC's `export default { ... }` gets a type error that `defineComponent` is the minimum-viable fix for

**Purpose:** Land 10 block-tag changes + one commit. Preserve blame history (SFCs aren't renamed — just their inner `<script>` tag attribute changes). Validate with Vite build + Vitest + vue-tsc (if installed by Plan 05 — not this plan's responsibility; this plan uses tsc with shims).

**Output:**
- 10 `.vue` files modified in place — `<script>` → `<script lang="ts">`
- Minimal type annotations inside each script body if tsc complains (most won't)
- `@ts-expect-error` budget ≤ 2 across all 10 SFCs
- 1 content commit: `refactor(ts): migrate 10 .vue SFCs to <script lang="ts">` (D-12-06 shape)
</objective>

**All 10 SFCs currently use Options API with a single `<script>` block:**
```vue
<template>
  ... HTML ...
</template>

<script>
// CommonJS or ES imports:
import Something from 'somewhere';
// OR the TitleBar-style CommonJS:
// const { ipcRenderer } = require('electron');

export default {
  name: 'ComponentName',
  components: { ... },
  props: { ... } OR props: [ 'title', 'ico' ],
  data() { return { ... }; },
  computed: { ... },
  methods: { ... },
  mounted() { ... },
  beforeUnmount() { ... },
};
</script>

<style lang="scss"> or <style scoped>
  ... CSS ...
</style>
```

**Conversion is surgical — just the opening `<script>` tag:**
```vue
<script>         →  <script lang="ts">
```

**The body of the <script> block can remain AS-IS in 95% of cases.** Options-API components with typed imports flow through TS without annotations (Vue's `defineComponent` inference works even without the wrapper, via plugin-vue's `<script lang="ts">` handling).
